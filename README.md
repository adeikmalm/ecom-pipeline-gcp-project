
# End-to-End E-Commerce Data Pipeline: From OLTP to Analytics Data Mart

## 1. Project Objective & Business Value
In the fast-paced e-commerce industry, raw transactional data generated by OLTP systems (often in nested JSON formats) is highly susceptible to anomalies and is not optimized for direct analytical querying. 

This project is a **Data Engineering Portfolio Showcase** demonstrating the design and implementation of a fully automated, scalable data pipeline using the **Modern Data Stack (MDS)**. The primary objective is to extract semi-structured operational data, enforce strict data quality rules, and transform it into a highly optimized multidimensional model (**Star Schema**) to deliver reliable Daily Sales Performance metrics for business stakeholders.

---

## 2. System Architecture
The architecture applies a decoupled approach, ensuring that data ingestion, storage, and transformation are handled by dedicated, purpose-built systems.

<img width="5206" height="5494" alt="architecture" src="https://github.com/user-attachments/assets/4da980cf-5377-4579-9d57-5b4722a9b71b" />

**Brief Workflow:**
1. **Source & Ingestion**: Real-time e-commerce data simulated via API is transacted into a **MongoDB** database. **Apache Airflow** periodically orchestrates the extraction of this data and loads it as raw JSON into the cloud data warehouse.
2. **Storage**: **Google BigQuery** acts as the central data warehouse, handling the raw landing zone.
3. **Transformation & Modeling**: **dbt Core** takes over within BigQuery to unpack nested arrays, cleanse anomalies, cast data types, and model the final Data Marts using the Medallion Architecture (Bronze, Silver, Gold).

---

## 3. Tech Stack Highlights
This project leverages the following core technologies to build a robust pipeline:
- **Python 3.11**: Custom data generation and API integration (`requests`, `pymongo`).
- **MongoDB**: Operational NoSQL database (OLTP) simulating the e-commerce backend.
- **Apache Airflow**: Workflow orchestration and automated data ingestion (deployed via Docker).
- **Google BigQuery**: Serverless Cloud Data Warehouse handling large-scale data processing.
- **dbt Core (v1.11)**: SQL-first transformation, data testing, and data modeling engine.

---

## 4. Data Modeling Strategy (Medallion Architecture)

The data flows through three distinct refinement layers to ensure analytical readiness:

### 🥉 Bronze Layer (Raw Data)
The immutable landing zone for raw data ingested directly by Airflow.
- `raw_carts`: Raw shopping cart transactions containing nested array items.
- `raw_users` & `raw_products`: Raw master data for customers and product catalogs.

### 🥈 Silver Layer (Staging & Cleansing)
The conformance layer where data is flattened, cleaned, and standardized using `Views`.
- **Nested JSON Flattening**: Utilized BigQuery's `UNNEST` to break down product arrays into granular transactional rows.
- **Data Cleansing**: Eliminated negative quantity anomalies, stripped currency symbols, and deduplicated records using `ROW_NUMBER()`.
- **Type Safety**: Enforced explicit `STRING` casting on all primary keys (`cart_id`, `user_id`, `product_id`) to prevent mathematical aggregation errors and ensure future UUID compatibility.

### 🥇 Gold Layer (Business Data Marts)
The presentation layer consisting of physical `Tables` optimized for Business Intelligence dashboards.
- **Dimensions**: `dim_users` and `dim_products` for descriptive analytical filtering.
- **Fact**: `dm_daily_sales_performance` aggregating key business metrics by joining transactions with master data.

---

## 5. Key Analytics Metrics Delivered
The final Star Schema exposes the following ready-to-use metrics for downstream BI tools (e.g., Power BI, Looker):
- **Total Revenue**: Cleansed financial accumulation of sold items.
- **Total Orders**: Volume of unique cart transactions.
- **Total Active Buyers**: Count of unique customers transacting per day.
- **Average Order Value (AOV)**: Revenue divided by total unique orders.

---

## 6. Engineering Highlights
- **Idempotent Design**: Ingestion pipelines are designed with `WRITE_TRUNCATE` dispositions for master data to ensure safe reruns (backfilling) without duplication.
- **Compute Optimization**: Handled complex one-to-many JSON relationships directly in the data warehouse using SQL, avoiding heavy Python-based transformations in memory.
- **Parallel Processing**: dbt executions are configured with multi-threading (`threads: 4`) to compile and build staging and mart models simultaneously, significantly reducing pipeline execution time.
